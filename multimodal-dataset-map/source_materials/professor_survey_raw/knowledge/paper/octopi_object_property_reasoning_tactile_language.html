Octopi: Object Property Reasoning with Large Tactile-Language Models

Samson Yu, Kelvin Lin, Anxing Xiao, Jiafei Duan, Harold Soh · NUS / University of Washington · RSS 2024 · arXiv:2405.02794 · PDF

One-liner. Octopi bolts a GelSight tactile encoder onto a Vicuna LLM (via a CLIP visual backbone + a LLaVA-style projection module) so that a vision-language model can feel — predicting hardness, roughness, and bumpiness from tactile video and then chaining those properties into commonsense reasoning (e.g., picking the ripe avocado because it feels softer) — trained on a new GelSight dataset, PhysiCleAR.

Problem & motivation

LVLMs reason about physical properties only from visual or text inputs, so they fail under visual ambiguity — material, texture, compliance, and occluded state often live in touch, not pixels. For humans, touch supplies physical information that vision cannot (material, temperature, compliance), improving physical reasoning. The authors want general-purpose robots, especially LLM-empowered ones, to do the same. The obstacle is a large domain gap: vision-based tactile sensors like GelSight produce image frames that are far from the natural images CLIP/LVLMs were trained on, and there is no dataset pairing GelSight tactile video with physical-property labels and language descriptions. The paper closes both gaps: a dataset (PhysiCleAR) and a tactile-language model (Octopi).

Method

Two contributions: the PhysiCleAR training/eval suite (covered under Setup) and the Octopi model (Fig 3).

Architecture. Three trained components: (1) a tactile input encoder, (2) a projection module, (3) an LLM. The tactile encoder is built on a CLIP ViT-L/14 visual encoder; following ViFi-CLIP, frame-level CLIP embeddings are average-pooled to a video-level representation. The projection module is two linear layers with an intermediate GELU activation (LLaVA-style), mapping tactile embeddings into Vicuna v1.5's word-embedding space. Two new word tokens <tact_start> and <tact_end> delimit a tactile frame sequence from a single sensor reading; the projected tactile embeddings are spliced between them and merged with ordinary word embeddings to form the instruction. Two LLM sizes: Octopi-7b (Vicuna-7b) and Octopi-13b (Vicuna-13b).

Three-step training.

(i) Encoder fine-tuning. Because tactile frames are a large distribution shift from natural images, the CLIP visual encoder is adapted via multitask physical-property classification. Using the ViFi-CLIP architecture, 8 task-specific learnable prompts (Visual Prompt Tuning) are attached to the input of each Transformer layer while the backbone is frozen; three separate classification heads predict hardness, roughness, and bumpiness, trained jointly with cross-entropy. The checkpoint that correctly predicts all three properties for the most objects is selected.

(ii) Tactile feature alignment. The classification heads are discarded; the fine-tuned encoder's outputs become the output embeddings. Only the projection module (and the two new token embeddings) is trained to align encoder outputs with language annotations; encoder and LLM are frozen.

(iii) End-to-end fine-tuning. The projection module plus LoRA adapters on the LLM are fine-tuned (encoder frozen) to improve coherence and similarity to the language annotations. LoRA: rank 128, scaling 256, dropout 0.05.

Inference / reasoning. At test time Octopi is instructed to describe the physical properties of a tactile reading, then to use those predictions for downstream reasoning (property comparison, superlative selection, object matching, scenario reasoning). The intermediate property descriptions are the bridge: the LLM reasons over its own emitted descriptions (e.g., "ripe avocados are moderately hard with small bumps") to answer the scenario.

Setup

  • Datasets / benchmarks: PhysiCleAR, a new GelSight dataset of 74 everyday objects, 408 tactile videos plus corresponding object videos, collected by hand with two exploratory procedures (pressing, then rotation for shear). Three physical properties, each 3-way: hardness (soft / moderately hard / hard), roughness (smooth / slightly rough / rough), bumpiness (no bumps / small bumps / big bumps); Table I. Annotated by 3 annotators (averaged), >1,200 annotations, high inter-annotator agreement (ICC3k 0.894 / 0.979 / 0.792). 80-10-10 split (60 train / 7 val / 7 test objects). Five tasks (Table III): Object Property Description (OPD), Property Comparison (PC), Property Superlative Selection (PSS), Property-object Matching (POM), and Property Scenario Reasoning (PSR, eval-only).
  • Hardware / simulator: GelSight tactile sensor for data collection. Real-robot demo: two GelSight sensors on a 7-DoF Franka Emika Panda for avocado ripeness classification (instructions relayed via ROS).
  • Baselines: Random baseline on every task; PG-InstructBLIP (physically-grounded VLM from visual input) on avocado property prediction; internal ablations (no-OPD, base vs fine-tuned CLIP, with/without LoRA, 7b vs 13b).
  • Compute: Encoder fine-tuning 30 epochs, <5GB VRAM, 6 h. Alignment + end-to-end: 5 h (7b) / 6.5 h (13b). 1× NVIDIA RTX A6000 (7b), 2× A6000 (13b).

Object-property prediction on unseen objects (Table IX, accuracy %): combined (all three correct) Octopi-7b 47.37, Octopi-13b 55.26, vs random 3.70 and fine-tuned-CLIP 57.89; per-property the 13b model reaches hardness 78.95, roughness 78.95, bumpiness 78.95.

Scenario reasoning (Table VII, PSR accuracy %): Octopi-7b 69.57, Octopi-13b 67.39, vs random 50.00; removing OPD drops 13b to 39.13 (below random) — direct evidence the intermediate property descriptions carry the reasoning.

Avocado real-robot (Table VIII): Octopi-13b property prediction 35.50% (random 3.70), ripeness classification 63.00% (random 50.00) over 10 avocados / 200 tactile samples / 100 pairs — and the visual baseline PG-InstructBLIP collapses on these avocados (hardness 37.50, roughness 3.00, bumpiness 9.50), always picking the first object on ripeness, since ripeness is hard to see but felt through softness.

Where it loses / surprises:

  • Octopi-7b beats the larger Octopi-13b on PSR (69.57 vs 67.39) — scale did not help reasoning here (the authors flag this as "marginal").
  • The fine-tuned-CLIP encoder slightly out-predicts Octopi end-to-end on raw combined property accuracy (FT-CLIP 57.89 vs Octopi-13b 55.26 in Table IX), i.e., wrapping the encoder in an LLM costs a little raw prediction in exchange for language reasoning.
  • Ablations confirm both fine-tuned CLIP (Tables X-XI) and LoRA end-to-end fine-tuning (Tables XII-XIII) are necessary: base-CLIP and no-LoRA variants degrade across tasks (e.g., 13b POM 60.43 → 34.76 without LoRA).

Limitations & open questions

From the authors:

  • Only three properties (hardness, roughness, bumpiness); geometric and affective properties deemed too hard for GelSight and excluded.
  • Data collected by hand to avoid sensor damage and because securing irregular objects is hard — limits scale and consistency of exploratory motion.
  • Future work: improve the tactile encoder, add more exploratory procedures, combine other tactile-sensor datasets, add robot proprioception, align with object images, and fine-tune the LLM on additional physical-understanding data.

What I noticed reading it:

  • Tiny test object set. Generalization is judged on 7 held-out objects (and avocado eval on 10 avocados). High per-property accuracies (~79%) rest on a very small object count — an n that small makes the accuracy numbers noisy and easy to overfit to.
  • Properties are 3-bucket categorical. Hardness/roughness/ bumpiness are coarse 3-way labels; the actual continuous compliance/texture signal a manipulation controller would need (force, friction coefficient) is quantized away — the reasoning never touches continuous force.
  • Pressing-only at deployment. Avocado readings use pressing only (no rotation), yet the model trained on press+rotate still works; the paper reads this as robustness, but it also means the shear/rotation signal may contribute less than the dataset design implies.
  • The LLM reasons over its own text, not the tactile latent. PSR collapsing below random without OPD shows reasoning is mediated entirely through emitted natural-language property descriptions — a fragile pipeline (a wrong property word poisons the downstream answer) rather than end-to-end tactile reasoning. This is a strength (interpretable) and a weakness (error-propagating).
  • No comparison to a tactile-classifier + symbolic-rule baseline; hard to tell how much the LLM adds over "predict 3 properties, look up a rule table."

Why I care

This is squarely on-theme for the thesis that many manipulation predicates are not visually evaluable — surface_is_rough, is_soft, is_ripe live in touch, not pixels. Octopi is a clean existence proof: a VLM that, given GelSight input, can name property-predicates (hardness/roughness/bumpiness) and then reason with them. Relative to BLADE, where predicates are learned as visual classifiers fθ(p): O → {T,F} over RGB-D, Octopi is the tactile analogue of the predicate-evaluation step: it grounds a small vocabulary of physical predicates in a non-visual modality. The natural question for my line: could a BLADE-style operator library use a tactile-language model like Octopi as the precondition/effect evaluator for predicates that vision can't check (is_grasped, is_inserted, surface_is_rough)? Octopi's most relevant limitation for me is that its reasoning is mediated by emitted text rather than a continuous force signal, and its predicates are coarse and categorical — exactly the gap between a property-describer and a planning-grade precondition checker. Off the abstraction-learning core, but directly load-bearing for the "touch-grounded predicates" extension of BLADE/PARL.

Newly ingested in the 2026-06-24 batch — directly relevant to this work:

  • TVL (A Touch, Vision, and Language Dataset) — sibling touch-vision-language dataset/model; the closest contemporaneous touch–language grounding effort to PhysiCleAR.

  • Touch100k and TLV — larger touch-language-vision datasets; what PhysiCleAR's property-labeled suite could scale toward.
  • Sparsh and T3 — self-supervised / transferable tactile representations; alternatives to Octopi's CLIP-fine-tuned tactile encoder for the (i) encoder step.
