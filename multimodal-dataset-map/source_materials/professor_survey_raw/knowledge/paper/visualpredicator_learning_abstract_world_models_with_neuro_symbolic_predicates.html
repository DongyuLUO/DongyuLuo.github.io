VisualPredicator: Learning Abstract World Models with Neuro-Symbolic Predicates for Robot Planning

Yichao Liang, Nishanth Kumar, Hao Tang, Adrian Weller, Joshua B. Tenenbaum, Tom Silver, João F. Henriques, Kevin Ellis · Cambridge / MIT / Cornell / Princeton / Oxford / Alan Turing Institute · arXiv 2024 (also published at ICLR 2025 per header) · arXiv:2410.23156 · PDF

One-liner. VisualPredicator introduces Neuro-Symbolic Predicates (NSPs) — first-order predicates implemented as Python snippets that can crop images and call VLMs — plus an online algorithm that interleaves VLM-driven predicate proposal, classification-accuracy-based predicate selection, and an extended cluster & intersect high-level-action learner, producing abstract world models that solve out-of-distribution long-horizon tasks across five simulated robot domains with near-Oracle sample efficiency.

Problem & motivation

Robot planning needs state abstractions that are simultaneously perceptually rich (e.g., "the spatula is greasy", "the jug is inside the coffee machine") and logically rich (e.g., "the number of blocks on each side of the balance is equal"). Traditional symbolic TAMP uses hand-coded predicates that can't adapt to new environments, while prior predicate-invention work (Silver et al. 2023, Konidaris 2018) restricts predicates to functions over hand-engineered object features — expressive enough for block-stacking but not for "is the jug inside the coffee machine" (an occlusion-heavy perceptual question) or "are the platters balanced" (a logical aggregation over counts). The paper argues for an abstraction language that is at once visually grounded (anything a VLM can perceive) and computationally rich (anything expressible in Python).

Method

Three contributions: (i) NSPs as a representation, (ii) an online algorithm that invents them, (iii) an extended operator learner.

1. Neuro-Symbolic Predicates (NSPs). An NSP is a Python function that returns a Boolean over a raw state and a tuple of objects. Two flavors:

  • Primitive NSPs evaluate directly on the raw observation. Their bodies can call a small Python API: get_object(type), get(obj, feature) for proprioceptive/object-pose features (Appendix A), crop_to_objects([o1, o2, ...]) to focus the image on the relevant objects, and evaluate_simple_assertion(text, image) to query a VLM with a natural-language assertion. Example: Holding(block) first checks gripper-fingers proprioception, then crops to the robot and block and asks the VLM "the block is held by the robot" (Fig 2).
  • Derived NSPs are computed from the truth values of other NSPs (analogous to PDDL axioms / derived predicates). OnPlate(block, plate) recursively checks if a block is DirectlyOnPlate or transitively on another block that is on the plate (Fig 2). Derived NSPs never appear in HLA postconditions because they can be re-evaluated from primitive truth values.

VLM accuracy boosters. Two tricks to improve primitive NSP precision: (a) condition VLM queries on the previous action, previous image, and previous truth value of the queried ground atom — resolves cases like "is the gripper holding the jug or merely next to it" by leveraging the prior Pick(jug) action (Appendix B.1); (b) overlay numeric ID labels on each object in the image (set-of-mark prompting, Yang et al. 2023) so the VLM can disambiguate "block2" from "block5".

2. Online predicate-invention loop (Algorithm 1). The agent receives an environment E, training tasks T, initial (largely empty) predicate set Ψ0, initial HLA set Ω0, and an empty interaction dataset. For up to nmax_iter rounds it: (1) Explore — runs the current planner/abstractions on training tasks until solved or planning budget nabstract exhausted, collecting positive transitions from successful skill executions and negative (state, action, FAIL) tuples from skill failures, plus satisficing plans when achieved. (2) Track best-so-far solve rate ρbest and best (lowest) number of infeasible plans νbest. If solve rate is stagnant, run Propose (§5.2), Select (§5.3), and Learn HLAs (§5.4).

3. Predicate proposal (three VLM-prompting strategies, §5.2). (a) Discrimination: show the VLM pairs of states where a skill succeeded vs. failed and ask it to write Python that predicts skill applicability — targets useful preconditions. (b) Transition modeling: show before/after snapshots of successful executions, ask for code describing what changed — targets postconditions. (c) Unconditional generation: ask the VLM for logical extensions (conjunctions, recursions) of existing predicates with no data conditioning — creates derived NSPs.

4. Predicate selection (§5.3). Most VLM proposals are low-precision, and overfitting to a too-large set is the failure mode. The authors define a novel objective: score a candidate set Ψ by (i) first learning HLAs Ω for Ψ via the extended operator learner, then (ii) measuring how accurately those HLAs classify transitions, plus a syntactic-simplicity bias. Greedy best-first search (GBFS) over predicate subsets, starting from goal predicates, takes minutes on a single CPU. After "enough" satisficing plans are collected, switch to the planning-efficiency-based objective from Silver et al. 2023.

5. Extended operator learner (§5.4 + Appendix B.2). Builds on Chitnis et al. 2022 cluster & intersect but replaces the restrictive "intersect over all feasible states" precondition with a minimum-complexity precondition that softly penalizes syntactic complexity: J(ω.PRE) = (positive coverage + negative exclusion) / |D| + α · |ω.PRE|. This produces optimistically loose preconditions, which the authors argue are essential for online exploration — restrictive preconditions inherited from demonstration-based learners prevent the planner from trying potentially-feasible skills it hasn't yet observed succeed.

Setup

  • Domains (5, PyBullet, Fig 3): Cover (pick & place blocks onto target regions; train 2 blocks / 2 targets, test 3/3); Blocks (build tower with Pick/Stack/PlaceOnTable; train 3-4 blocks, test 5-6); Coffee (jug-into-machine, MakeCoffee, Pour into cups; jug may need rotation; 5 skills; train 1 cup, test 2 or 3); Cover Heavy (Cover variant with white marble blocks that are physically too heavy — "impossible tasks" that the agent must detect as unreachable); Balance (PressButton activates a machine only when blocks on both sides of a balance beam are equal; train 2 or 4 blocks, test 4 or 6).
  • Training: 5 training tasks per domain (10 in Cover Heavy and Balance, 20 in Blocks). Planning budget nabstract = 8 (100 for Coffee). 5 random seeds.
  • Baselines (6): MAPLE (HRL, Nasiriany 2022 — learns Q function over behavior primitives, no explicit predicates; tested only on training distribution because it can't generalize to more objects); ViLa (Hu et al. 2023, zero-shot VLM action plans, open-loop); Sym. pred. (same online loop but predicates restricted to object-feature functions; closest in representation to Interpret (Han et al. 2024) but learns from interaction, not human instruction); Ablate op. (no min-complexity preconditions, plain cluster-&-intersect); No invent (use only the initial under-specified abstractions); Oracle (manually designed predicates and operators).
  • Compute: single CPU for most baselines; MAPLE uses a GPU.

Results

Headline (Fig 4, mean over 5 seeds): VisualPredicator matches the Oracle across Cover, Blocks, Coffee, Cover Heavy, and Balance — near-perfect solve rate on test tasks that involve more objects / out-of-distribution goal counts than training. Specific findings:

  • Q1 (vs. HRL/VLM): MAPLE struggles even on in-distribution tasks under matched interaction budget; ViLa generates infeasible plans (e.g., picks a jug without considering orientation, repeatedly tries to pick already-held blocks) consistent with the LLM-planning critique of Kambhampati et al. 2024 (cited; not yet ingested).
  • Q2 (vs. Oracle and No-invent): VisualPredicator decisively beats No invent, showing predicate invention is necessary; matches Oracle, showing the invented abstractions are as effective as manually-designed ones.
  • Q3 (vs. Sym. pred.): Plain symbolic predicates over object features handle simple Cover but fail in Coffee (can't determine JugInMachine from object poses alone — occlusion), Cover Heavy (can't perceive "too heavy"), and Balance (can't reason about block counts without derived predicates). This is the central evidence that NSPs' VLM-querying + Python-logic combination is doing real work.
  • Q4 (vs. Ablate op.): Without the min-complexity operator learner the system overfits early (e.g., requires JugInMachine as a precondition of Pour), yielding good train but poor test performance. The extended operator learner is what enables exploration-based learning to generalize.

Limitations & open questions

From the authors (Appendix B.5, §8 Conclusion):

  • No recovery mechanism for failed plans — an obvious extension flagged in the conclusion.
  • Exploration efficiency could be improved.
  • Scaling to partially observable and real-world domains is future work; all five experiment domains are fully observed simulation.
  • Relaxing the assumption of pre-given skills (using RL or policy synthesis, citing Liang 2024, Ma 2023, Huang 2024) is open.
  • VLM accuracy is not 100% on simple primitive queries; the previous-action / set-of-mark tricks help but don't eliminate failures.

What I noticed reading it:

  • The min-complexity precondition objective is a small but important methodological contribution that deserves emphasis — it's the online-learning analog of cluster & intersect and may transfer to other operator-learning settings.
  • VLM cost per predicate query is not reported (number of VLM calls per training round, per skill, per task) — relevant for assessing whether NSPs are tractable at larger predicate counts.
  • Predicate names are still VLM-proposed in code; how often semantically-equivalent predicates are proposed (synonyms, near-duplicates) and how the system deduplicates is not detailed.
  • All experiments are fully observed; the same VLM-conditioning trick (previous action + previous image) hints at how to extend to partial observability but isn't tested under occlusion regimes harder than Coffee.
  • Comparison to demonstration-based predicate invention (BLADE, Migimatsu & Bohg) is implicit only — the paper positions itself in the online-from-interaction regime, not demonstration-from-language, leaving a clean axis for future head-to-head comparison.

Why I care

This is the direct predecessor to ExoPredicator — same first author Yichao Liang, same Cambridge/MIT/Cornell crew, same NSP representation, same online predicate-invention paradigm. ExoPredicator extends VisualPredicator from purely endogenous world models (robot effects only) to handle exogenous processes (boiling water, machines that change state on their own). Reading VisualPredicator is prerequisite to understanding what ExoPredicator inherits vs. extends.

Other connections worth tracking:

  • Cluster sibling to BLADE (BLADE): both auto-invent first-order abstractions for bilevel planning, but BLADE learns from language-annotated demonstrations while VisualPredicator learns from online exploration with no demos. The two paradigms (demo-from-language vs. interaction-from-scratch) are direct alternatives — a topic page comparing their trade-offs would be high-value. Notably, BLADE's biggest stated limitation is precisely predicate invention, which VisualPredicator solves — if the two could be married, the result would invent predicates from demos + online interaction rather than picking one.
  • Methodological lineage: builds directly on Silver et al. 2023 predicate invention (the predicate-set scoring framework), Chitnis et al. 2022 cluster & intersect (operator learning — extended here with the min-complexity objective).
  • Sibling work (same first author): PoE-World — another Liang+Tenenbaum+Ellis line on program-structured world models, complementary perspective on how to organize learned abstractions.
  • Pixels-to-predicates lineage: From Pixels to Predicates works the same axis — VLMs as the perceptual back-end for symbolic abstractions — useful for the topic page on VLM-grounded predicate learning.
  • For my own work: the min-complexity precondition objective and the three predicate-proposal strategies (discrimination / transition modeling / unconditional generation) are reusable methodological primitives. The discrimination strategy in particular looks generically useful for any "what makes this skill apply" learning problem.

Broadly intelligent agents should form task-specific abstractions that selectively expose the essential elements of a task, while abstracting away the complexity of the raw sensorimotor space. — Abstract
The core of our approach is to learn an abstract model of the environment in terms of Neuro-Symbolic Predicates (NSPs), which are snippets of Python code that can invoke vision-language models (VLMs) for querying perceptual properties, and further algorithmically manipulate those properties using Python, in the spirit of ViperGPT and VisProg. — §1, Introduction
Our precondition learner ensures that each data in the transition dataset is modeled by one and only one high-level action and minimizes the syntactic complexity of the HLA to encourage optimistic world models. — §5.4, Learning High-Level Actions
While [Sym. pred.] performs well in simple domains like Cover, it struggles to invent predicates that require grounding on perceptual cues not explicitly encoded in object features. For instance, in Coffee, it cannot reliably determine if a jug is inside a coffee machine based on object poses — a key precondition for the TurnMachineOn action. — §6, Results and Discussion (Q3)
