Inner Monologue: Embodied Reasoning through Planning with Language Models

Wenlong Huang, Fei Xia, Ted Xiao, Harris Chan, Jacky Liang, Pete Florence, Andy Zeng, Jonathan Tompson, Igor Mordatch, Yevgen Chebotar, Pierre Sermanet, Noah Brown, Tomas Jackson, Linda Luu, Sergey Levine, Karol Hausman, Brian Ichter · Robotics at Google / Stanford · CoRL 2022 · arXiv:2207.05608 · PDF · project page

One-liner. Inner Monologue closes the LLM-planning loop by continually injecting natural-language feedback — success detection, passive/active scene description, and human responses — into the planner's prompt as the robot executes, enabling the frozen LLM to replan, retry, and self-correct without any task-specific finetuning.

Problem & motivation

Prior LLM-as-planner work (SayCan, Huang et al. "Language Models as Zero-Shot Planners" [20]) is essentially one-directional: the LLM emits a sequence of skill calls, and the agent executes them open-loop or with affordance gating that doesn't communicate back into the prompt. That breaks under stochastic skill failure, dynamic environments, occlusion, and ambiguous instructions. The paper's question: how far can grounded textual feedback alone — no extra training, no new architecture — carry LLM planners in real embodied settings? They frame this as an "inner monologue" analogous to humans verbalizing intermediate observations and choices while solving multi-step tasks.

Method

Setup. A frozen pretrained LLM is the planner. The agent has a library Π of pretrained short-horizon skills πk with short language descriptions ℓk (trained via RL or BC). At each step the LLM sees the human instruction i, the running history of executed actions, and textual feedback o appended to the prompt, and emits the next skill call. No finetuning; few-shot prompting only.

Three feedback channels (Fig 2):

  • Success Detection — binary task-specific signal from an engineered detector (sim) or learned classifier (real) reporting whether the last skill succeeded. Surfaced in the prompt as "Success: True/False".
  • Passive Scene Description — structured, automatically-injected scene information at every planning step. Subdivided into Object feedback (open-vocab object recognition output) and Scene feedback (task-progress summary; in the tabletop domain the LLM itself infers achieved sub-goals as a chain-of-thought).
  • Active Scene Description — unstructured feedback only produced when the LLM asks, answered by a VQA model or by a human. Lets the agent gather targeted info or user preferences (in this paper the active answer is always human-provided, called Human feedback).

Three concrete instantiations (one per domain; Fig 3):

  • Simulated tabletop rearrangement (Ravens-based): InstructGPT planner + scripted Object/Success/Scene modules + CLIPort-like pick-and-place primitive. Chain-of-thought used for Object+Scene variant.
  • Real tabletop rearrangement: InstructGPT planner + MDETR open-vocab detector for Object and Success + scripted suction pick-place.
  • Real kitchen mobile manipulation (Everyday Robots): PaLM planner + SayCan-style value-function affordance grounding + learned success classifier (Success) + human-provided object recognition (Object) + optional Human channel for active queries.

The unifying recipe is purely prompt-level: feedback strings are interleaved in a fixed convention (e.g., [Robot action, Scene, Robot thought]) and the LLM's next emission is parsed as the next skill call. Replanning is implicit — the LLM simply re-decodes given the updated context.

Setup

  • Datasets / benchmarks: Ravens-based simulated tabletop (4 seen + 4 unseen tasks, 50 episodes each, with injected Gaussian disturbances at test time); a real UR5e tabletop pick-place setup (3-block stacking, sort fruits-vs-condiments, 10 runs each with injected 4 mm Gaussian action noise clipped at 2σ); real-kitchen mobile manipulation on Everyday Robots, reusing the SayCan task suite (manipulation, drawers, mobile manipulation; 120 evaluations with and without adversarial disturbances).
  • Hardware / simulator: Ravens-style simulator; UR5e arm with wrist-mounted RealSense RGB-D; Everyday Robots mobile manipulator in a real office kitchen.
  • Baselines: CLIPort (with and without termination oracle) on the simulated tabletop; LLM-only variants ("+LLM Object") that drop the Success and Scene channels; SayCan on the kitchen mobile manipulator.
  • Compute: not reported. Planners are InstructGPT and PaLM, used frozen via prompting.

Results

Simulated tabletop (Table 1, 50 episodes each). On unseen tasks under test-time disturbances, CLIPort scores 0% across the board; LLM+Object alone hits 0–62%; adding Success pushes to 56–76%; the full Object+Scene variant tops at 26–86%. Headline: Object+Scene on "Put the blocks on mismatched bowls" goes 0% (CLIPort) → 86% (Inner Monologue). On seen tasks Object+Scene wins on 3 of 4 categories, hitting 94% on "Pick and place" vs CLIPort+oracle's 74%.

Real tabletop (Table 2, 10 runs each). Finish 3-block stacking: 20% (LLM Object) → 100% (Inner Monologue Object+Success). Sort fruits from bottles: 20% → 80%. Both feedback signals are needed: Success alone misses initially-occluded objects (initial scene desc gap); Object alone can't recover from policy noise without retry signals.

SayCan collapses to ~0% on adversarially-disturbed manipulation/mobile manipulation; Inner Monologue recovers because the LLM sees the failure string and either retries or selects a new skill. Fig 4 shows that LLM planning failures and control failures both drop as more feedback channels are added.

Emergent capabilities (Fig 5, qualitative): continued adaptation to mid-rollout instruction changes ("nevermind, I want you to finish your previous task"); self-proposing new goals under infeasibility (after "the purple block is too heavy", the agent proposes "find a lighter block"); multilingual interaction (Chinese instruction parsed and replanned into English actions); interactive scene QA over the action history. All of these appear without being demonstrated in the prompt. Inner Monologue is also reported as robust to feedback-order swaps and typos in the human instruction.

Limitations & open questions

From the authors (Sec 5):

  • In the kitchen and tabletop instantiations the scene/object feedback is partly oracle-provided (human or scripted). Learned-perception feasibility is studied only in Appendix Table 5.
  • Failure modes: false-negative success detection causes wasted retries; false-positives introduce adversarial partial observability that the LLM can't recover from. Sometimes the LLM ignores feedback and emits skills for objects not actually present.
  • The whole stack is bottlenecked by the underlying low-level policies — better LLM reasoning can't expand the achievable task set beyond what skills can execute.
  • No mechanism for expressing uncertainty over the feedback channels (e.g., low-confidence detections enter the prompt as flat strings).

What I noticed reading it:

  • "Replan" here is implicit re-decoding from an updated prompt — there's no separate plan repair stage. That's elegant but means the method inherits all of the LLM's prompt-sensitivity, and there's no explicit commitment / no-regression guarantee.
  • Real-world Object feedback in the kitchen domain is human-provided, which arguably blurs the line between "Passive Scene Description" and the "Human" active channel.
  • Skill library is fixed; the LLM cannot propose a new primitive when the existing library is insufficient (cf. Voyager, which does exactly this).
  • Aggregate numbers on the simulated tabletop are dominated by the Scene channel, which on that domain is the LLM summarizing its own progress — effectively chain-of-thought re-labeled. That's a weaker form of "environment feedback" than the success/object signals.

Why I care

This is the canonical reference for closed-loop LLM planning via natural-language feedback, and the direct intellectual ancestor of the replan-after-feedback pattern BLADE adopts at the symbolic level. The mapping is clean:

  • Inner Monologue's Success channel ↔ BLADE's check that the expected effects (effa) became true after policy rollout, triggering replan if not.
  • Inner Monologue's Object/Scene channel ↔ BLADE's predicate classifiers being re-evaluated each step before the planner is re-invoked.
  • Inner Monologue's Active/Human channel ↔ BLADE doesn't have an analogue; a natural extension would be active queries when the classifier confidence is low or the planner can't find a satisfying sequence.

The key differentiator: Inner Monologue does all of this in prompt on a flat skill library, while BLADE does it over a learned PDDL operator library with explicit precondition/effect symbols. The "structure-as-prior vs scale-everything" debate (see BLADE's Why-I-care) maps directly here. For future work, the Self-Refine sibling line is the LLM-only analogue (LLM critiques its own output rather than environment text); for the open-ended skill-library extension, see Voyager; and for the code-emission alternative to flat skill calls, see Code as Policies.

Quotable

We propose that by leveraging environment feedback, LLMs are able to form an inner monologue that allows them to more richly process and plan in robotic control scenarios. — Abstract
In contrast, Inner Monologue studies settings where grounded environment feedback is provided directly to the LLM in a closed-loop fashion. This promotes improved LLM reasoning in complex long-horizon settings, even before any external affordance-based grounding methods are applied. — §3.2
Without any LLM-informed feedback SayCan has success rate close to 0% since it does not have explicitly high-level retry behavior. Inner Monologue significantly outperforms SayCan because of its ability to invoke appropriate recovery modes depending on the environment feedback. — §4.3 Analysis
Although LLMs can generate fluent continuation from the prompted examples, we surprisingly find that, when informed with environment feedback, Inner Monologue demonstrates many impressive reasoning and replanning behaviors beyond the examples given in the prompt. — §4.4
