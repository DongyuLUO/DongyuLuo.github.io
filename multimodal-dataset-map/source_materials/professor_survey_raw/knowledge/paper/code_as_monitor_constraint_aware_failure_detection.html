Code-as-Monitor: Constraint-aware Visual Programming for Reactive and Proactive Robotic Failure Detection

Enshen Zhou*, Qi Su*, Cheng Chi†, Zhizheng Zhang, Zhongyuan Wang, Tiejun Huang, Lu Sheng†, He Wang† · Beihang University / Peking University / BAAI / Galbot · CVPR 2025 · arXiv:2412.04455 · PDF

One-liner. Code-as-Monitor (CaM) recasts both reactive and proactive open-set failure detection as spatio-temporal constraint satisfaction: a VLM writes monitor code over compact geometric “constraint elements” (points, lines, surfaces) abstracted from the scene, and the code — evaluated cheaply every frame via a tracker rather than re-querying the VLM — returns a boolean + reason that triggers replanning, yielding real-time closed-loop correction.

Problem & motivation

Robots in long-horizon tasks fail unpredictably, so closed-loop systems need to detect failures. The paper distinguishes two modes (following Crandall et al. [29]): reactive detection identifies a failure after it occurs (the lobster has landed on the table — a delivery failure), and proactive detection prevents a foreseeable failure (a tilting pan will cause the lobster to fall, so correct the tilt now). Both are hard in the open-set regime where failures are not predefined. Prior VLM-based detectors treat this as visual question answering (VQA): they query the VLM/LLM repeatedly during execution, which is slow, coarse-grained in 3D spatial reasoning, and (per the paper) inadequate for the high-precision, high-frequency monitoring proactive detection demands — e.g. catching a 2 cm displacement or a 180° rotation. Proactive open-set detection in particular is barely explored; prior proactive work [2, 10] is confined to predefined failures.

Method

CaM is three modules (Fig 2): a Constraint Generator, a constraint Painter, and a Monitor.

1. Constraint Generator. Given the global task instruction Lglobal, multi-view RGB-D observations, the previous subgoal, and any failure feedback from the Monitor, a VLM (GPT-4o) generates the next subgoal lnext plus its associated textual constraints — split into Cd (constraints that must be maintained during subgoal execution, e.g. “pan handle must be grasped,” “pan should remain horizontal during transfer”) and Cu (constraints that must hold upon completion, e.g. “pan directly above the stove”). This split is exactly how the authors unify reactive (a during/completion constraint already violated) and proactive (a during-constraint trending toward violation) detection.

2. Constraint Painter. For each textual constraint, the Painter abstracts the relevant entities/parts into compact constraint elements — sets of 3D points representing geometric primitives (points, lines, surfaces). The pipeline (Fig 3): a trained multi-granularity constraint-aware segmentation model (ConSeg) produces both instance-level masks and fine-grained part-level masks plus an element type description for each constraint × view; these masks are projected to 3D, voxelized (voxel size set by element type, e.g. a surface needs ≥3 points), clustered, and one representative point per voxel is filtered down to a target count, then connected per instance mask into the element e. End-effector points (fingertips, hand center) come directly from forward kinematics, bypassing perception. Elements are aggregated into E = {e0, …, en+k} and numerically annotated with unique colors/labels across all views. ConSeg itself is built on LISA [27] (a LLaVA VLM + SAM encoder/decoder) with LoRA fine-tuning; it does two inference passes (instance-level, then part-level) and emits a <SEG> token plus a textual element-type description.

3. Real-time Monitor. The annotated multi-view images OE serve as visual prompts: GPT-4o is given them plus the constraints and next subgoal, and writes monitor code — a function that reads the elements’ current 3D positions, performs NumPy arithmetic on them (e.g. computes the angle between the pan-surface normal and the world z-axis, or the distance between bread-points and pan-points), and returns False + a reason string when a tolerance is exceeded (e.g. pan surface tilt > 15°). Crucially, after generation the elements are tracked frame-to-frame with an off-the-shelf tracker (Co-Tracker [24]), so the monitor code can be re-evaluated every frame at ~1 Hz without calling the VLM again — this is what makes high-precision real-time monitoring affordable. On False, execution halts immediately and the reason string becomes the feedback fpre fed back to the Generator for re-planning; otherwise the subgoal completes and the cycle repeats.

Setup

  • Datasets / benchmarks: Three simulators — CLIPort [57] (Stack in Order, Sweep Half the Blocks), Omnigibson [31] (Slot Pen, Stow Book, Pour Tea), RLBench [22] — plus a real-world setting (Simple Pick & Place, Reasoning Pick & Place in cluttered scenes). ConSeg is trained on BridgeData V2 [64]: 10,181 trajectories / 219,356 images, with GPT-4o decomposing instructions into subgoals/constraints/object associations and Grounded SAM [53] + Semantic SAM [32] giving instance/part masks, refined into LISA-format training data. Segmentation evaluated on ReasonSeg [27] and a proposed ConstraintSeg benchmark (gIoU / cIoU).
  • Hardware / simulator: CLIPort uses a UR5 with suction cup + spatula (pre-trained CLIPort policy); Omnigibson uses a Fetch with gripper (ReKep [21] policy); RLBench uses a Franka + gripper (ARP [73] policy); real-world uses a UR5 with a Leap Hand [55], controlled by an open-loop DexGraspNet 2.0 [72] grasping policy. ConSeg fine-tuned per-simulator on 100 trajectories each (not fine-tuned for real-world, to test generalization). Frames captured at 1 Hz.
  • Baselines: DoReMi [16] (DRM) in all settings (frequent VQA-based intermediate-failure detection); Inner Monologue [20] (IM) in CLIPort (detects failures only at subgoal completion); ReKep [21] alone in Omnigibson; DexGraspNet 2.0 (DGN) alone in real-world. Segmentation baselines: OVSeg, GRES, X-Decoder, SEEM, PixelLM, LISA-13B, and a Foundation-Model-Combination (FMC = GPT-4o + Grounded SAM + Semantic SAM).
  • Compute: not reported (uses off-the-shelf GPT-4o and Co-Tracker; ConSeg is a LoRA-tuned LLaVA/SAM stack).

Results

Headline: across simulators CaM achieves 28.7% higher success rate and 31.8% lower execution time than baselines under severe disturbances.

  • CLIPort (Table 1). Under the most severe disturbance (p=0.3, q=3) in Stack in Order, CaM hits 88.33% vs DoReMi’s 76.67% (+17.5% abs.); on Sweep Half the Blocks CaM reaches 75.00% — ~4.5× DoReMi (16.67) — because counting block-points in a designated surface region beats VLM image-counting. Execution time on Stack-in-Order (p=0.15) drops to 21.0s vs IM 34.8 / DRM 26.0 (38.7% / 19.2% faster).
  • Omnigibson (Table 2). CaM beats ReKep and DoReMi on Slot Pen, Stow Book, and Pour Tea under point-, line-, and surface-level disturbances. In Pour Tea, only CaM detects surface-level failures (teapot pitch/roll changes) that DoReMi’s single-image VQA misses (DoReMi 0% on some Pour Tea disturbances). CaM cuts execution time 34.8% and token count 52.2% vs DoReMi by monitoring tilt angle in real-time instead of frequent VLM calls.
  • Real-world (Tables 3–4). On Simple Pick & Place with a Leap Hand, CaM surpasses DoReMi by ~20.4% across object types (deformable, transparent, small-rigid, large-geometric). On long-horizon Reasoning Pick & Place in clutter, only CaM closes the loop around the open-loop DexGraspNet policy (e.g. 60% vs 10% on “clear all objects except animals”), and dynamically re-targets when a human moves an object mid-task.
  • Segmentation (Table 5). ConSeg matches LISA/PixelLM on ReasonSeg but beats all baselines on ConstraintSeg by ~40% at the part level (60.2 gIoU / 65.3 cIoU part-level).
  • Ablations (Tables 4, 6, 7). Multi-view raises Stow-Book success 40%→65% (single-view causes occlusion + line/point misinterpretation); constraint-aware segmentation beats DINOv2 semantic points (65% vs 42.5%); pre-formed elements beat unconnected 3D points (65% vs 55%). Using constraint elements rather than raw entities lifts real-world success 20%→60% — raw entities give inaccurate/slow 3D tracking and resist arithmetic in code.

Limitations & open questions

From the authors:

  • The system depends on the VLM (GPT-4o) generating correct constraints and correct monitor code; the paper notes no standard exists for evaluating proactive-detection accuracy, and they deliberately do not report a failure-judgment success rate because the code runs continuously in real-time (Sec 4.1, Task Settings).
  • ConSeg used in real-world is not fine-tuned and the training data is limited to pick-and-place tasks; generalization to tool-use, articulated objects, and long-horizon complex tasks is validated qualitatively but the distribution gap is acknowledged.
  • Tracking quality (Co-Tracker) and accurate 3D element formation are explicit bottlenecks — the ablations show unconnected/raw-entity points degrade both tracking and in-code arithmetic.

What I noticed reading it:

  • Every constraint is geometric and vision-derived: angles, distances, and counts over 3D point sets projected from RGB-D. There is no pathway for a constraint like “water has come to a boil,” “surface became slippery,” or “pan is hot enough” — these have no visual geometric signature and no non-visual sensor feeds the monitor. The teaser’s “careful not to let the lobster drop” is reduced to a pan-tilt angle, not a thermal/contact state.
  • The constraints are authored fresh per subgoal by GPT-4o, not learned. There is no library of reusable, grounded change-predicates with preconditions/effects; the “abstraction” is a per-subgoal code snippet, semantically opaque and not composable across tasks the way a symbolic operator is.
  • “Temporal” here means per-frame re-evaluation of a static geometric predicate over a tracked element — spatio-temporal combinational dynamics — not modeling a state transition over time (no notion of “became X after action Y”).
  • The whole loop hinges on a closed-vocabulary-free but ultimately GPT-4o-bounded constraint proposal; the same scaling/consistency questions that haunt LLM operator proposal apply, untested past these task suites.

Why I care

CaM is the cleanest existing instantiation of the closed-loop monitoring axis of the dynamic-state-change idea: a perception module evaluates time-varying constraints during execution and uses violations to drive semantic replanning — exactly the reactive+proactive correction loop I want, and a sharper failure-detection mechanism than BLADE’s replan-after-each-behavior (BLADE replans on completion; CaM monitors continuously and can abort mid-behavior). The unification of reactive and proactive detection as one constraint-satisfaction formulation, split into “maintain-during” vs “hold-on-completion,” is a framing worth borrowing.

But it sits on the far vision-only side of my idea and contradicts two of its commitments. (1) Vision-only, geometric constraints: every monitored predicate is a distance/angle/ count over RGB-D-projected points. The dynamic concepts I care about — “hot enough now,” “slippery after wetting,” “coming to a boil” — live in tactile/thermal/audio signals that CaM has no channel for; a tilting-pan proxy is the closest it gets, and it is a geometric stand-in, not a sensed state. (2) VLM-authored code, not learned multimodal change-predicates: CaM’s constraints are per-subgoal GPT-4o programs, whereas BLADE-style and my target work want learned, grounded, composable predicates with effects over time. So CaM is the right loop architecture wrapped around the wrong sensing and representation — an excellent foil: “take CaM’s reactive+proactive constraint loop, but replace VLM-on-pixels geometric constraints with learned multimodal (tactile/audio/thermal) state-change predicates.” The contrast pairs naturally with REFLECT (multimodal-but-retrospective explanation) and Inner Monologue (textual closed-loop feedback): CaM is the most precise, real-time, but most narrowly-visual of the three.

  • [2, 10] Alvanpour / Diehl & Ramirez-Amaro — predefined-failure proactive-detection prior work CaM positions against.

Related papers already in the corpus:

  • BLADE — the anchor; CaM is a sharper continuous-monitoring failure detector vs BLADE’s replan-on-completion, but BLADE has learned symbolic predicates where CaM has VLM-authored geometric code.
  • REFLECT — multimodal failure explanation + correction, but retrospective; CaM is real-time/proactive but vision-only.
  • Inner Monologue — textual closed-loop replanning; CaM is the code/geometry-grounded, continuous-monitoring counterpart.
  • VisualPredicator — neuro-symbolic predicates from a VLM; contrast with CaM’s non-learned per-subgoal code constraints.
  • Multimodal-sensing contrasts for the “non-visual constraint” gap: Octopi, Tactile-VLA, Audio-VLA — the sensing channels CaM’s monitor lacks.
