REFLECT: Summarizing Robot Experiences for Failure Explanation and CorrecTion

Zeyi Liu, Arpit Bahety, Shuran Song  ·  Columbia University  ·  CoRL 2023  ·  arXiv 2306.15724  ·  arxiv.org/abs/2306.15724  ·  PDF  ·  project page

One-liner. REFLECT converts raw multisensory robot observations (RGB-D, audio, proprioception) into a three-level hierarchical text summary, then queries an LLM progressively to detect, localize, and explain failures — and optionally generate a correction plan — without any task-specific training.

Problem & motivation

Long-horizon robot tasks fail inevitably. Current systems either dump low-level execution logs (unreadable to humans and LLMs alike) or rely on task-specific failure detectors. The gap: no general framework existed that (1) converts unstructured, multimodal robot sensory data into a form an LLM can reason over, and (2) systematically asks the LLM "what went wrong and when" before proposing a fix. REFLECT addresses this gap with a domain-general, training-free pipeline built on the observation that a good robot summary must be both multisensory (visual, audio, contact each reveal different failure modes) and hierarchical (coarse subgoal-level alignment check first, fine-grained event-level detail on demand).

Method

The framework has three modules:

1. Hierarchical Robot Summary (§3.1)

Three abstraction levels are constructed from raw sensory streams:

  • L0 — Sensory-Input Summary. Each RGB-D frame is processed with an object detector (MDETR) to produce a 3-D semantic point cloud; object states are identified via CLIP cosine similarity; spatial relations (inside, on top of, near, …) are computed heuristically. These are assembled into a task-informed scene graph containing only task-relevant objects and their neighbors. Audio sub-clips are classified against a label bank via AudioCLIP embeddings. The result is one scene graph + audio label per timestep.
  • L1 — Event-Based Summary. Key frames are selected when the scene graph changes, an audio event fires, or a subgoal ends. Each key frame is rendered as a structured text caption: [timestep] Action: … Visual: obj [state], … obj is [relation] obj … Auditory: …. This is the detailed evidence layer the LLM reads when explaining a failure.
  • L2 — Subgoal-Based Summary. One caption per subgoal (the end-frame state), used to quickly verify whether each subgoal was achieved before diving into details.

2. Progressive Failure Explanation (§3.2)

The LLM (GPT-4) is queried in two stages:

  1. Success verification. For each subgoal, the LLM is given the L2 summary and asked "Is the subgoal satisfied?" (Yes/No). This quickly localizes the failure to a subgoal.
  2. Failure explanation.
    • If a subgoal fails: retrieve the L1 event-based summary up to that timestep and ask the LLM to explain what happened and what caused it (execution failure path).
    • If all subgoals pass but the task still fails: provide the original robot plan + final environment state and ask what is wrong with the plan (planning failure path).

The progressive structure means expensive detailed context is only loaded when needed, and the LLM's temporal reasoning is anchored to a structured representation rather than raw video.

3. Failure Correction Planner (§3.3)

Given the failure explanation and final state, the LLM generates a high-level correction plan (list of action primitives). Each generated action is mapped to its closest executable primitive via sentence-embedding similarity (following Huang et al. SayCan), making the plan directly executable.

Setup

  • Dataset: RoboFail — created alongside the paper. Simulation: 100 failure scenarios in AI2THOR across 10 tasks (execution and planning failures, including wrong action order, missing actions, ambiguous references, perception errors). Real-world: 30 failure scenarios with a UR5e arm across 11 tasks, with Intel RealSense D415 (RGB-D) and RODE VideoMic Pro+ (audio).
  • Hardware / simulator: AI2THOR for simulation; UR5e robot arm in a toy kitchen for real world.
  • Perception: MDETR (object detection), CLIP (object state + audio-language matching), AudioCLIP / Wav2CLIP (audio classification), GPT-4 (LLM).
  • Baselines: BLIP2 caption (replace scene graphs with BLIP2 image captions), w/o sound, w/o progressive (open-ended Socratic query, no hierarchical structure), subgoal only (L2 only, no L1), LLM summary (LLM summarizes L0 text directly), w/o explanation (correct without explaining).
  • Compute: not reported

Results

Three metrics: Exp (human-judged explanation correctness & informativeness), Loc (failure timestep localization accuracy), Co-plan (correction plan task success rate).

Simulation (Table 1): REFLECT achieves Exp=88.4, Loc=96.0, Co-plan=79.1 on execution failures and Exp=84.2, Loc=80.7, Co-plan=80.7 on planning failures — best across all conditions. The next-best single ablation is "Subgoal only" (Exp~76, Co-plan~75). Removing the explanation step ("w/o explanation") drops Co-plan to 41.9 / 56.1, demonstrating that explanation causally aids correction. Removing audio drops Exp and Loc by ~20 pp on execution failures.

Real world (Table 2): REFLECT: Exp=68.8, Loc=93.8. Performance is slightly lower than simulation, attributed to perception noise (object detection, state classification errors). BLIP2 caption is the worst baseline (Exp=6.25 simulation, 35.7 real), since BLIP2 captions lack spatial relations and object states that are critical for failure identification (Fig. 5 contrast).

Key ablation findings:

  • Audio is crucial: "w/o sound" loses ~20 pp Exp on execution failures because dropped objects and toggled appliances are easier to detect auditorily than visually.
  • Progressive querying helps: "w/o progressive" drops Loc substantially — flat open-ended queries don't anchor the LLM to specific timesteps.
  • Hierarchical structure matters: "Subgoal only" misses intermediate events (e.g., detecting a drop mid-subgoal), so L1 detail is essential for execution failures.
  • LLM-generated flat summaries lose information: "LLM summary" condition omits key failure-relevant details (e.g., "fridge was closed") that the structured L1 caption preserves.

Limitations & open questions

  • Scene graph heuristics may fail in more complex, cluttered environments — spatial reasoning trained on robotics data or fine-tuned VLMs could replace the hand-crafted rules.
  • Object state detection requires a predefined candidate list; a free-form LLM query for possible states would be more flexible.
  • The environment is assumed to be mostly static during execution; dynamic environments or moving objects would break the point-cloud aggregation.
  • Low-level control failures (joint slip, grasp force errors) are not captured — richer proprioceptive or tactile sensing would be needed.
  • Localization for planning failures is harder (no single failure timestamp), and accuracy drops compared to execution failures.
  • Open question: can the hierarchical summary structure be learned end-to-end rather than engineered?

Why I care

This paper directly addresses how to make LLM planners self-correcting — a key ingredient for long-horizon manipulation. The progressive explanation algorithm is a clean example of structured prompting with hierarchical context retrieval that avoids flooding the LLM with irrelevant low-level detail. The task-informed scene graph is related to the planning-abstraction representations (symbolic state, object relations) that underpin BLADE and predicate-learning work. The RoboFail dataset is a rare evaluation resource for failure-mode diversity in manipulation. For Weiyu's work on online replanning and correction, REFLECT's three-level summary architecture is a concrete design to compare against or build on.

Quotable

"We hypothesize that an effective failure reasoning framework requires several key components: first, a component to summarize 'what happened' by converting unstructured, multimodal robot sensory data into a structured, unified format; second, a component to reason 'what was wrong' by inferring from the summary whether expected outcomes of the robot plan were achieved; and finally, the ability to plan 'what to do' based on the failure reasoning to correct the failure and complete the task." — §1 Introduction, p.1
